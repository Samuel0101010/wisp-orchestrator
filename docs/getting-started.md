# Getting started with WISP

This is the hands-on companion to the [README](../README.md). By the end of
this guide you'll have WISP running locally, a multi-agent team assembled
from a built-in template, and a small TypeScript utility shipped end-to-end
through the orchestrator. Expect ~5 minutes of setup plus a 1–3 minute run.

## 1. What you'll build in this guide

A four-file TypeScript package (`kebab-case.ts` + vitest tests +
`package.json` + `README.md`) generated by a WISP team — architect, developer,
QA — running unattended under your supervision in the browser. You'll see the
live execution graph, the per-agent trajectories, and the verification gates
in action.

## 2. Prerequisites

- **Node.js** ≥ 20.10 (22 LTS or 24 LTS recommended — a prebuilt `better-sqlite3` binary ships for them, so no C++ compiler is needed)
- **pnpm** — optional; the launcher uses `corepack` (bundled with Node) if pnpm is not on `PATH`
- **git** installed and on `PATH`
- The **`claude` CLI** on `PATH`
- Either a **Claude Max** subscription (default — credentials are inherited
  from `~/.claude/`) **or** an `ANTHROPIC_API_KEY` if you prefer API mode
  (set `WISP_AUTH_MODE=api`)
- ~5 minutes wallclock + ~$0.10–$0.30 in API costs if you're on API mode
  ($0 on subscription)

Sanity-check the toolchain before you start:

```sh
node --version    # >= v20.10 (22 or 24 LTS recommended)
pnpm --version    # optional — corepack provides pnpm if this is missing
claude --version  # any recent version
git --version
```

## 3. Install (~2 min)

The plugin install is the supported path. From any shell:

```sh
claude plugin marketplace add Samuel0101010/wisp-orchestrator
claude plugin install wisp@wisp-local
claude /wisp-dashboard
```

The first time `/wisp-dashboard` runs after a fresh install, it auto-runs
`pnpm install && pnpm build` for you (1–2 minutes — uses `pnpm` if on PATH,
otherwise `corepack`, which is bundled with Node).
After that, the launcher boots straight to the dashboard server, picks a
free port in the `4400–4500` range, and opens your default browser.

> If the first command fails with `Permission denied (publickey)`, `plugin
> marketplace add` is cloning over SSH; on a machine with no GitHub SSH key,
> run `git config --global "url.https://github.com/.insteadOf" "git@github.com:"`
> once (keep the space between the two quoted args in PowerShell) and re-run it.
> See [troubleshooting](./troubleshooting.md) for details.

**Developer install** (if you cloned the repo locally):

```sh
pnpm install
pnpm build
WISP_SERVE_WEB=1 node apps/dashboard-server/dist/server.js
# then open http://127.0.0.1:4400
```

## 4. Open Mission Control

Once the dashboard is up at `http://localhost:4400`, you'll land on
**Mission Control**. The left sidebar holds primary navigation; on a fresh
install you'll see roughly these items (exact labels are i18n-driven — see
the Dashboard for the canonical names):

- **Mission Control** — KPIs and live runs
- **Team Chat** — conversation surface for the active run
- **Agents** — registry of available roles
- **Skills** — installed plugin skills
- **Workers** — real-time view of each agent's subprocess
- **Insights** — trajectories, costs, replan history
- **GOAP Planner** — interactive DAG canvas
- **Prompt Bundles** — cached prompts for re-use
- **Settings** (bottom of the sidebar) — theme, language, data clearing

On first launch the project list is empty. Click the **"+ New project"**
button in the sidebar to start.

## 5. Create your first project

Fill in the New Project dialog:

- **Name:** `hello-wisp`
- **Repo path:** any existing git-initialized directory you don't mind WISP
  writing to. The orchestrator creates per-task **worktrees** under
  `<repoPath>/../worktrees/`, so prefer an empty scratch repo for your first
  run:
  ```sh
  mkdir hello-wisp && cd hello-wisp && git init
  ```
  Paste the absolute path of that directory into the field.
- **Template (optional but recommended):** pick `ts-library` from the
  template picker. This seeds the team with architect / core-dev / test-dev
  / qa and pre-fills a suitable goal.

Click **Create**.

## 6. Write a goal

If you picked the `ts-library` template, the goal field is already populated.
Otherwise paste this:

```
Create a small TypeScript utility "kebab-case.ts" that exports a single
function kebabCase(input: string): string converting any string to
kebab-case. Include 5 unit tests covering: simple words, camelCase,
PascalCase, snake_case, and mixed input. Use vitest. Set up
package.json with type: module, vitest as dev dep, and a test script.
Provide a README.md showing usage.
```

Click **Generate plan**. The planner agent (opus) reads your goal + team
spec and emits a DAG with roughly 3–5 task nodes (typical shape: setup →
implementation → tests → docs, plus an auto-injected runtime-verifier on
terminal nodes).

## 7. Review the team and plan

The **GOAP Planner** view opens automatically once the plan is generated.
You'll see the DAG rendered as a React Flow canvas. Click any node to open
its side panel, where you can edit:

- the per-task **prompt**
- **dependencies** on other nodes
- **success criteria** (which build / test / lint commands gate completion)
- **maxTurns** budget

If you used the `ts-library` template, every task already has an agent
assigned — the picker is in the side panel's "Agent" slot.

Need to swap an agent? Open **Team Chat** (or the TeamBuilder route) and
edit the role's `model`, `allowedTools`, or `systemPrompt`. Saves take
effect on the next plan generation.

## 8. Run it

When the plan looks right, click **Lock & Run** at the top right of the
plan editor. The plan transitions from `draft` to `locked` and the Walker
begins dispatching tasks.

Switch to the **Workers** view to watch the agents work in real time. Each
running task shows:

- the role + model
- the worktree branch
- streaming NDJSON events from `claude -p`
- token / turn counters as they tick

**Expected runtime:** 1–3 minutes for the kebab-case task on a healthy
connection.
**Expected cost:** ~$0.10–$0.30 in API mode, $0 in subscription mode.

Verification gates run automatically when a subprocess exits — `build`,
`test`, `lint`, plus optional runtime-verify (see README §
*Runtime verification*). A failed gate triggers one retry with the failure
tail embedded in the next prompt; a failed retry marks the task `failed`
and skips its descendants.

## 9. Inspect the output

When the run finishes:

- **Team Chat** — each agent's full transcript, scoped to the run.
- **Insights** — per-agent trajectories (which agents ran how long), cost
  summary, replan badge if QA forced a replan.
- **Your repo** — the produced files live on the last successful worktree
  branch. To collect them into your main branch:
  ```sh
  cd /absolute/path/to/hello-wisp
  git branch --list   # find the run's final task branch
  git checkout <task-branch>
  ```
  The `/wisp-inspect` slash command also surfaces the result branch + a
  git-log graph of the run.

Pause the run, resume after rate-limit cooldown, or kill it from the
RunView controls at any time. Snapshots are written every 10 minutes under
`${WISP_DATA_DIR}/snapshots/` and a restart of the server recovers the run
state automatically.

## 10. Next steps

- Try the **`refactor-squad`** template on an existing TypeScript repo to
  see WISP work against a non-empty codebase.
- Add custom **success criteria** on the project detail page (Definition of
  Done card) — for example a `smoke` HTTP probe, an `e2e` Playwright check,
  or a `manual` sign-off gate. See README § *Runtime verification*.
- **Customize a team** in TeamBuilder, then click *Save as Template* to
  reuse it (templates persist under `<WISP_DATA_DIR>/templates/`).
- Read [docs/architecture.md](architecture.md) to understand how the
  Walker, SubprocessPool, and verification gates fit together.
- Browse [docs/templates.md](templates.md) for the full catalog of built-in
  team templates.
- For deeper development workflows (mock CLI, Drizzle migrations,
  WebSocket debugging) see [docs/development.md](development.md).
